## ストーリー構成アプローチ（選択肢）

| アプローチ | 説明 | 本プロジェクトへの適合 |
|---|---|---|
| **Feature-Based（推奨）** | 機能（F-01〜F-11）を軸にストーリーを整理 | 要件が機能単位で定義済み。コード生成との対応がとりやすい |
| User Journey-Based | 「ヒント入力→執筆→印刷」の流れでストーリーを整理 | 体験フローが明快。ただし機能境界が曖昧になりやすい |
| Persona-Based | 児童ストーリー群・保護者ストーリー群に分離 | ペルソナが 2 種類のみのため Feature-Based と大差なし |
| Epic-Based | Epic（大きな機能塊）配下にサブストーリーを配置 | Phase 0/1/2 の段階的展開に対応しやすい |

ユーザーストーリーをどの軸で整理しますか？

B) **User Journey-Based** — 「ヒント入力 → 執筆 → 印刷」の体験フロー軸で整理。UX の流れが伝わりやすい
C) **Epic-Based** — Phase 0（デモ）/ Phase 1（MVP）を Epic として、サブストーリーを配置
X) Other (please describe after [Answer]: tag below)

児童ユーザーのペルソナをどう切り分けますか？

A) **1ペルソナ（全学年統一）** — 「小学生〜高校生」をひとつのペルソナとして扱う。シンプルで実装優先
B) **2ペルソナ（低学年 / 中高学年）** — 低学年（小1〜3）と中高学年（小4〜高3）に分ける。将来の低学年規格対応を見越した設計
C) **3ペルソナ（小低・小高・中高）** — より細かく分ける（Phase 2 対応が多い場合に有用）
X) Other (please describe after [Answer]: tag below)

Given / When / Then 形式を採用します。
この形式は受け入れ基準が明確になり、テストやコード生成に

## INVEST 基準（ストーリー品質チェック）

生成するストーリーはすべて以下を満たします：

| 基準 | 内容 |
|---|---|
| **I**ndependent | 他ストーリーと独立して実装・テスト可能 |
| **N**egotiable | 実装方法に柔軟性がある |
| **V**aluable | ユーザーまたはビジネスに具体的な価値を提供 |
| **E**stimable | 規模感が見積もり可能 |
| **S**mall | 1 スプリント（または短期間）で完了可能 |
| **T**estable | 受け入れ基準が明確で検証可能 |
